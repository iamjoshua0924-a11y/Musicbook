# 코드위키 수동 모드 확장 및 독립 검증 – 작업계획서

## 0. 문서 목적 / 범위
본 문서는 현재 인프라(Express + Socket.io + MongoDB) 및 뷰어(PDF + 주석 Fabric.js)를 확장하여 **[코드위키 모드]**를 추가하기 위한 단계별 작업계획을 정의한다.

- **Phase 1**: *독립 샌드박스 알고리즘 검증* (프로덕션 코드/서버 로직 변경 금지)
- **Phase 2**: *프로덕션 서버 결합 및 UI 이식* (Phase 1 컨펌 후 진행)

> 중요(보안/컴플라이언스): “Cloudflare/캡차/봇 차단 무력화” 등 **보안 우회 목적의 구현은 진행하지 않는다.**  
> 다만, **정상적인 자동화(테스트/사용성)** 범위 내에서 puppeteer를 사용해 동작을 검증할 수는 있으나, 차단을 우회하는 stealth/탐지회피/우회 헤더 조작을 “원천 우회” 수준으로 설계하는 것은 제외한다.  
> 대상 사이트 정책에 따라 (1) 공식 API/허용된 임베드/제휴, (2) 사용자가 제공한 HTML/텍스트 입력, (3) 서버 캐싱 및 요청 빈도 제한 등 **합법적·안전한 접근 방식**을 우선 채택한다.

## 1. 요구사항 요약

### 1.1 UI/기능
- 뷰어에서 **[구글 PDF 모드] / [코드위키 모드]**를 사용자가 수동 선택
- 코드위키 모드에서 입력한 외부 URL을 파싱해 **코드-가사 정렬(공백/줄바꿈 포함) 100% 보존**
- 일본어(한자/카나) → **한국어 독음** 변환 후에도 “수직 정렬”이 깨지지 않도록 렌더링
- 세션(룸)에서 페이지 넘김 대신 **스크롤 백분율 동기화** (마스터→참여자)
- 텍스트 악보 컨테이너 위에 **주석 레이어(Fabric.js) 스크롤 동기화** (좌표 어긋남 방지)

### 1.2 데이터 포맷
`LogBlock[]` (예: 1글자/공백 단위의 “수직 레고 블록” 구조)
```json
[
  { "chord": "AM7", "lyric_kr": "라쿠(楽)" },
  { "chord": "", "lyric_kr": "엔(園)" },
  { "chord": "", "lyric_kr": " " },
  { "chord": "B", "lyric_kr": "나" }
]
```

## 2. Phase 1 – 독립 샌드박스 알고리즘 검증 (test-parser.js)

### 2.1 목표(Definition of Done)
- **프로덕션 서버/클라이언트 코드를 변경하지 않고**, 프로젝트 루트에 `test-parser.js`(또는 `test-parser.mjs`)만 추가
- `node test-parser.js <url>` 실행 시:
  - 외부 URL로부터 악보 원문 텍스트를 가져오고(가능한 경우)
  - 줄바꿈/공백을 유실하지 않고 파싱해
  - `LogBlock[]`를 `console.dir(blocks, { depth: null, maxArrayLength: 200 })` 등으로 보기 좋게 출력

### 2.2 구현 항목(샌드박스)
1) **스크래핑/가져오기 모듈**
   - 1차: `fetch`로 HTML 가져오기(가장 단순/안전)
   - 2차(필요 시): `puppeteer` 기반 “렌더링 후 DOM 추출”
   - 요청 제한/캐싱/리트라이(과도 요청 방지)
   - 실패 시 에러/원인(권한/차단/구조변경) 명확히 로그
   - **Fair Use / 정상 범위 내 호환성 확보(중요)**
     - 목적: “차단/우회”가 아니라, **동적 렌더링 사이트에서 Headless 기본값으로 발생할 수 있는 렌더링 누락/크래시를 줄이기 위한 호환성 설정**만 적용
     - 적용 범위(최소치):
       - 안정적인 viewport 지정(예: 1365×768 수준)
       - locale/Accept-Language 지정(예: `ko-KR` 또는 사용자 선택)
       - **표준 Chrome User-Agent 지정(호환성 목적)**  
         > 주의: 이는 탐지 회피(stealth)나 캡차/차단 무력화를 위한 것이 아니며, 사이트가 접근을 거부/차단하는 경우에는 우회하지 않고 실패로 처리한다.
       - 네트워크 idle 대기(`networkidle*`) 후 DOM 추출(과도 대기 금지: 타임아웃 설정)

2) **본문 추출(Extract)**
   - 사이트별 규칙을 “플러그인” 형태로 분리(ChordWikiExtractor, UGExtractor …)
   - 최소 요구: 본문을 “코드 라인”과 “가사 라인”이 포함된 **텍스트 형태**로 추출
   - 원본 공백/탭/개행은 그대로 유지 (정렬 무결성)

3) **파서(Parse) – 1:1 수직 블록화**
   - 입력: “코드 라인(상단)” + “가사 라인(하단)” 또는 interleaved 형태
   - 출력: `LogBlock[]`
   - 규칙:
     - 공백(스페이스/탭)도 블록으로 유지
     - 코드 위치는 “해당 인덱스 위치”에 chord를 배치(없으면 빈 문자열)

4) **독음 변환(Translate / Reading)**
   - 1차(현실적/품질): 외부 사전/라이브러리(예: kuromoji 형태소 분석 + reading 변환) 검토
   - 2차(간단 MVP): 카타카나/히라가나 → 한글 표기 규칙(간이 로마자/한글 치환) + 일부 한자 대응표(부분)
   - 산출물은 “원문(옵션) + 독음” 형태로 확장 가능:
     - `lyric_raw`, `lyric_kr`

### 2.3 산출물
- `test-parser.js`
- (옵션) `parsers/` 폴더(사이트별 extractor) – *Phase 1에서는 파일 1개만 요구되므로, 내부 모듈은 파일 내로 시작 후 Phase 2에서 분리*

### 2.4 리스크/대응
- 사이트 차단/구조 변경: extractor 실패 가능 → “사용자 입력(원문 붙여넣기)” 폴백 설계(Phase 2에서 지원)
- 독음 품질: 100% 정확한 한자 독음은 외부 사전/엔진이 필요 → MVP/고도화 단계 분리

## 3. Phase 2 – 프로덕션 결합 및 UI 이식 (Phase 1 컨펌 후)

### 3.1 서버(API) 설계
1) **프록시 API**
`GET /api/proxy-chord?url=...`
- 응답: `{ ok: true, blocks: LogBlock[], meta: {...} }`
- 보안:
  - 허용 도메인 allowlist(ChordWiki, Ultimate Guitar 등)
  - 요청 rate limit(사용자/세션 단위)
  - 결과 캐싱(짧은 TTL)
  - URL 검증(SSRF 방지: private IP/localhost 차단)

2) **렌더링 수집(Puppeteer) 폴백**
- 서버에서 plain `fetch`로 403/429/봇 검증 페이지가 감지되면, **Puppeteer(headless)로 1회 폴백**하여 “실제 브라우저와 유사한 컨텍스트”에서 HTML을 렌더링 후 본문을 추출한다.
- Fair Use를 위해 이미지/폰트/미디어 요청은 차단하고(document/xhr/script 위주), 필요한 최소 데이터만 수집한다.
- 주의: “탐지회피(stealth)” 목적의 우회는 하지 않으며, 렌더링 호환성/오탐(False Positive) 감소를 위한 UA/언어/viewport 정도만 최소 적용한다.

#### 배포 환경 주의(Puppeteer)
- 본 구현은 `puppeteer-core`를 사용하므로 Chromium 바이너리가 필요하다.
- 권장: Docker 배포 시 Dockerfile에서 `chromium` 설치 + `PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium` 설정

2) **세션 스크롤 동기화 이벤트**
- 마스터(페이지터너)가 `scroll` → `session:scroll:sync { roomCode, fileIdOrDocId, ratio }`
- 서버는 룸 브로드캐스트, 팔로워는 부드럽게 `scrollTo`
- 디바운스(예: 50~120ms) 및 변경 폭 임계치(예: 0.2% 이상) 적용

### 3.2 프론트(UI/렌더링) 설계
1) **모드 토글 UI**
- 링크 입력창 옆에:
  - [구글 PDF 모드]
  - [코드위키 모드]
- 코드위키 모드에서 URL 입력 후:
  - PDF canvas 영역 숨김
  - 텍스트 악보 렌더 영역 표시

2) **정렬 렌더링(Alignment Shift 방지)**
- 렌더 기준은 `LogBlock[]`의 “셀 단위”
- CSS:
```css
.chord-sheet { font-family: ui-monospace, SFMono-Regular, Menlo, Monaco, Consolas, "Noto Sans Mono", monospace; }
.chord-line { display:flex; align-items:flex-start; gap:0; }
.chord-lyric-cell { display:flex; flex-direction:column; min-width:max-content; }
```
- 독음이 길어지면 셀 폭이 늘어나며, 상단 코드/하단 가사가 함께 밀려 **수직 정렬 유지**

3) **스크롤 동기화(코드위키 모드에서)**
- 마스터 scroll 이벤트에서 `ratio = scrollTop / (scrollHeight - clientHeight)` 계산
- 팔로워는 동일 ratio로 scrollTop을 계산해 `scrollTo({ top, behavior: 'smooth' })`

4) **주석 레이어(스크롤 연동)**
- 텍스트 컨테이너 전체 높이(스크롤 높이) = Fabric 캔버스 높이로 매핑
- 스크롤 시:
  - 텍스트와 캔버스가 함께 이동하도록 “같은 스크롤 컨테이너에 포함”
  - 또는 캔버스는 absolute overlay + translateY(scrollTop) 방식
- 기존 “페이지 기반” 주석 모델과 충돌 가능성이 있으므로,
  - 코드위키 모드는 `docId(=url hash)` 기반 “단일 문서 캔버스”로 저장/동기화 설계(SSOT 재정의)

## 6. 실제 작업절차(8단계) – “지시 즉시 착수”용 실행 설계
아래 8단계는 본 계획서의 내용을 “실제로 구현 완료”까지 가져가기 위한 실행 절차이다.  
각 단계는 **완료조건(DoD)** 과 **산출물** 이 명시되어 있어, 사용자가 “N단계 시작”이라고 지시하면 즉시 착수할 수 있다.

### 6.1 (1단계) Phase 1 준비 – 샌드박스 뼈대
**목표:** 프로덕션 코드/라이브 서버 로직을 건드리지 않고, 테스트만 실행 가능한 최소 스크립트 기반을 만든다.
- 작업
  - 프로젝트 루트에 `test-parser.js`(또는 `test-parser.mjs`) 단독 생성
  - CLI 인자 처리: `node test-parser.js <url>` 형식
  - 에러 로깅 표준화(HTTP status, selector not found 등)
- 완료조건(DoD)
  - URL 미입력 시 사용법 출력
  - URL 입력 시 “가져오기 시도 → 성공/실패” 흐름 로그가 명확
- 산출물
  - `test-parser.js` 1개 파일

### 6.2 (2단계) Phase 1 – 가져오기(Extract) 구현(안전 우선)
**목표:** 외부 페이지에서 악보 본문 텍스트를 “가능한 범위 내에서” 추출한다.
- 작업
  - 1차: `fetch`로 HTML 수집 → 본문 후보 selector 기반 추출(사이트별)
  - 2차(필요 시): `puppeteer`로 렌더 후 DOM 텍스트 추출(단, 우회/stealth 목적 금지)
  - 호환성 목적의 최소 설정(렌더링 누락/크래시 방지):
    - viewport/locale/Accept-Language 지정
    - 표준 Chrome User-Agent 지정(호환성 목적)
    - `networkidle` 대기 + 명확한 타임아웃
  - 과도 요청 방지(간단 rate limit/재시도/타임아웃)
- 완료조건(DoD)
  - 최소 1개 대상 URL에서 “코드+가사 텍스트(공백/개행 포함)” 추출 성공
- 산출물
  - `test-parser.js` 내 extractor(사이트별 분기 포함)

### 6.3 (3단계) Phase 1 – 정렬 보존 파서(LogBlock) 구현
**목표:** 공백/개행 100% 보존 + 코드 위치/가사 위치를 1:1 블록(LogBlock)으로 변환한다.
- 작업
  - 코드 라인/가사 라인(또는 interleaved 형태)을 입력으로 받는 파서 구현
  - 공백도 블록으로 유지
  - 코드 위치는 인덱스(컬럼) 기준으로 해당 위치 블록에 chord를 주입
- 완료조건(DoD)
  - `LogBlock[]` 출력으로 원문 재구성이 가능한 수준(공백 포함)
  - chord가 원래 위치에서 밀리지 않음(샘플 검증)
- 산출물
  - `parseAndTranslateJpop(url): LogBlock[]`의 parse 파트

### 6.4 (4단계) Phase 1 – 일본어→한국어 독음 변환 연결(MVP)
**목표:** 일본어 가사에 한국어 독음을 제공하되, 정렬(수직 일치)은 절대 깨지지 않게 한다.
- 작업
  - MVP: 가나 변환 + 간단 매핑(부분)
  - 확장: 형태소/사전 기반 reading 변환(가능 시, 추후)
  - 출력 필드: `lyric_kr` (옵션으로 `lyric_raw` 유지 가능)
- 완료조건(DoD)
  - 독음이 길어져도 “블록 단위”라 데이터 정렬 무결성 유지
  - 예시 JSON 포맷으로 출력 가능
- 산출물
  - `parseAndTranslateJpop()` 완성

## 7. 추가 UX/품질 개선 백로그(차후 작업)
아래 항목들은 코드위키 기능과 무관하게 “노래책/뷰어” 사용성 개선을 위한 백로그이다.  
**본 섹션은 계획만 작성하며, 사용자가 “착수”를 지시하기 전까지 구현하지 않는다.**

### 7.1 가능보컬 필터: 모달 기반 멀티선택(검색 + 칩) (추천)
**배경:** 가능보컬 유저가 40명 이상으로 늘어날 경우, 체크박스 나열 방식은 UI가 지저분해지고 찾기 어렵다.

**목표(DoD):**
- 검색 영역에는 “가능보컬(AND) · N명 선택됨” 버튼/요약만 노출
- 버튼 클릭 시 모달(또는 드롭다운 패널) 오픈
  - 상단 검색 input
  - 하단 유저 리스트(스크롤) + 체크
  - 선택된 유저는 상단에 칩으로 표시(X로 제거)
- 선택된 유저 전원이 가능한 곡만 필터링(AND) 로직 유지

**구현 포인트(예상):**
- 파일: `public/musicbook/index.html`, `public/musicbook/musicbook.js`, `public/musicbook/musicbook.css`
- 데이터: 기존 `/api/availability/users`, `/api/availability?userId=` 재사용
- 성능: 선택된 유저만 availability를 fetch하고, 캐시(Map<userId,Set>) 유지

### 7.2 “가능곡 편집” 진입 시 필터/정렬이 바뀌는 문제 – 필터 로직 일원화
**현상:** “가능곡 편집”을 누르면, 사용자가 보던 필터/정렬이 유지되지 않거나 정렬이 바뀌는 느낌이 든다.  
**원인 후보:** 일반 모드(카드 단위)와 편집 모드(파일 단위)가 서로 다른 데이터셋/정렬 기준을 사용.

**목표(DoD):**
- 편집 모드 진입 전의 필터 상태(검색어/장르/분위기/보컬/가능보컬/정렬필드/정렬방향)가 **편집 모드에서도 동일하게 적용**
- 편집 모드 종료 시에도 사용자가 보던 필터 상태 복구

**정리 방향(안):**
- 공통 “필터 상태”를 하나로 정의(예: `state.filters = { q, genre, mood, vocal, availableVocalIds, sortField, sortDir }`)
- `applySongFilters()` 내부에서
  - 카드 리스트/파일 리스트 각각에 “동일한 필터/정렬 규칙”을 적용하도록 분기만 유지
  - `createdAt`/latest 정렬은 “숫자(ms)” 기준으로 통일

### 6.8 (8단계) Phase 2-3 – 세션 스크롤 동기화 + 주석 레이어 결합
**목표:** 코드위키 모드에서는 페이지가 아닌 “스크롤 위치(%)”를 동기화하고, 주석도 함께 스크롤되며 좌표가 어긋나지 않게 한다.
- 작업
  - Socket 이벤트: `session:scroll:sync { roomCode, docId, ratio }`
  - 마스터(페이지터너)만 송신, 팔로워는 smooth scroll
  - 코드위키 모드의 주석 SSOT를 `docId(=url hash)` 기반 단일 문서 모델로 재정의
  - 텍스트 컨테이너 scrollHeight와 Fabric 캔버스 높이를 1:1로 매치
- 완료조건(DoD)
  - 마스터 스크롤이 팔로워에게 동일 비율로 동기화됨(튐/지연 최소)
  - 주석이 스크롤 후에도 위치 어긋남 없음
  - PDF 모드(페이지 기반) 주석/세션과 충돌 없음
- 산출물
  - socket 핸들러 + viewer 렌더/주석 결합 코드
